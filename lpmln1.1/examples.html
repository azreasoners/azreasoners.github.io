Example LPOD Programs

This section comprises of some domains that we have tested the lpmln2asp program on.

  • abcd Example


    •                         
a >> b :- not c.
b >> c :- not d.

  • hotel Example


    •                         
dom(1..3).

1{hotel(X): dom(X)}1.
:- hotel(1), not close.
:- hotel(1), not star2.
:- hotel(2), not med.
:- hotel(2), not star3.
:- hotel(3), not tooFar.
:- hotel(3), not star4.

close >> med >> far >> tooFar.

star4 >> star3 >> star2.

  • abc Example


    •                         
dom(1..2).

1{a(X): dom(X)}2.
1{c(X): dom(X)}2.

% b is true iff a is false
b(X) :- dom(X), not a(X).
:- a(X), b(X).

a(X) >> b(X) :- c(X).

  • university Example


    •                         
% We need to choose a university and a major from 3 universities and 3 majors: computer science & engineering (major(1)), electrical engineering (major(2)), and industrial engineering (major(3)). 

dom(1..3).

:- univ(X), univ(Y), dom(X), dom(Y), X!=Y.
:- major(X), major(Y), dom(X), dom(Y), X!=Y.

% We have a preference on majors
major(1) >> major(2) >> major(3).

% We have a specific preference on the majors for each university

major(1) >> major(3) >> major(2) :- univ(1).
major(2) >> major(3) >> major(1) :- univ(2).
major(2) >> major(1) >> major(3) :- univ(3).

% We know the ranking of the universities

univ(3) >> univ(1) >> univ(2).

% We know the ranking of the universities for each major

univ(3) >> univ(1) >> univ(2) :- major(1).
univ(1) >> univ(2) >> univ(3) :- major(2).
univ(2) >> univ(3) >> univ(1) :- major(3).

% There are some majors in some universities that our applications were rejected
:- univ(1), major(1).
:- univ(3), major(1).
:- univ(3), major(2).

    Benchmark Examples and Results

    This section comprises of the benchmark examples that we have tested the lpmln2asp program on. Note that we include the atom "dom(*)" in the body of some regular ASP rules because they are necessary in the input program to the original implementation of LPOD -- , and we want our comparison in the following tables to be fair.

  • N_hotel Example

    • This example contains 3 LPOD rules with n atoms in the heads. The benckmark for N_Hotel example is automatically generated with a parameter n, which denotes the number of candidate hotels and determines the complexity of each LPOD rule. Below is a generated program when n = 10. 

                            
% We need to choose a hotel from n hotels. We randomly assign the order of their price, distance, and service.

1{hotel(X): dom(X)}1.

price(X) :- hotel(X), dom(X).
distance(X) :- hotel(X), dom(X).
service(X) :- hotel(X), dom(X).

:- hotel(X), price(Y), X!=Y, dom(X), dom(Y).
:- hotel(X), distance(Y), X!=Y, dom(X), dom(Y).
:- hotel(X), service(Y), X!=Y, dom(X), dom(Y).

dom(1..10).

price(9) >> price(1) >> price(6) >> price(5) >> price(10) >> price(3) >> price(4) >> price(8) >> price(2) >> price(7).
distance(4) >> distance(6) >> distance(10) >> distance(8) >> distance(2) >> distance(7) >> distance(1) >> distance(3) >> distance(5) >> distance(9).
service(10) >> service(5) >> service(1) >> service(7) >> service(6) >> service(4) >> service(2) >> service(9) >> service(3) >> service(8).

    The run-time comparison of finding Pareto-preferred answer sets of this program is shown below where we use the original LPOD implementation as our baseline.

    n = PSMODELS LPOD2ASPRIN
    10 0.060s 0.438s
    50 2.485s 1.961s
    60 4.825s 1.788s
    70 Terminate with No Result 3.147s
    100 Terminate with No Result 6.730s
    200 Terminate with No Result 47.868s
    300 Terminate with No Result 1m45.006s

    As we see, our system is much more tolerant to increasing the complexity of LPOD rules than the original LPOD implementation. Besides, the original LPOD implementation cannot find any answer set if the number (n) of atoms in the head of an LPOD rule exceeds 70 whereas our system can find the preferred answer set even when n=800 (800 is not an upper bound for our system, but is just the biggest number we've tested).

  • N_abc Example

    • This example contains n LPOD rules with 2 atoms in the heads. To run the following benchmark, please add "-c n=NUMBER" in your command line to define the value of n, where NUMBER is replaced by any positive integer. 

                            
dom(1..n).

1{a(X): dom(X)}2.
1{c(X): dom(X)}2.

% b is true iff a is false
b(X) :- dom(X), not a(X).
:- a(X), b(X).

a(X) >> b(X) :- c(X).

    The run-time comparison of finding Pareto-preferred answer sets of this program is shown below where we use the original LPOD implementation as our baseline.

    n = PSMODELS LPOD2ASPRIN

    15 10.088s 11.374s

    As we see, our system is much more tolerant to increasing the number of LPOD rules than the original LPOD implementation.

    Furthermore, it is known that using preference statements, some fragment of LPOD can be succinctly represented in the language of asprin via the translation into Answer Set Optimization (ASO). We manually translate N_abc program into an ASO program, which is then translated into the language of asprin, including the component, the component, and the component. With these three programs, one can find the Pareto-preferred answer sets of N_abc program following the instructions in . However, the reduction under inclusion preference requires a translation from LPOD to ''ranked'' ASO programs, which further requires a more complex reduction to the language of asprin. Besides, the reductions from LPOD to ASO programs under cardinality and penalty-sum preferences were not shown. In comparison, our method reduces LPOD directly to the language of asprin, which yields a simpler and uniform method that applies to all preference criteria for LPOD. Our experiments show that our method is more efficient than the method via the reduction to ASO even for Pareto preference.
